Below is a concrete **V1 technical spec** for the geometry-only phase of the product: **single input photo with a printable in-frame reference board → rectified image + scale metadata**.

This spec assumes someone else will implement it, so it is written as an engineering handoff, not a brainstorm.

---

# V1 Technical Spec: Single-Photo Reference-Board Rectification Engine

## 1. Goal

Build an **on-device, single-photo rectification engine** that takes a handheld photo of a flat pattern piece placed beside a **printed reference board** and returns a geometrically corrected top-down image suitable for later segmentation and vectorization.

V1 is **geometry only**. It does **not** need to detect or segment the pattern outline.

The engine must:

* detect a known printable reference board in the image
* estimate the plane shared by the reference board and pattern piece
* compute a fronto-parallel rectified view of that plane
* recover metric scale from the known printed board geometry
* emit outputs and metadata for downstream processing

The implementation target is a **Rust core library** with a **CLI frontend**. The design must support later reuse in WebAssembly and native mobile apps.

---

# 2. Core Product Decision

V1 should **not** require per-user camera calibration.

Instead, V1 should use a **known printable reference board placed in the same photo and same plane as the pattern piece**.

This is the right V1 tradeoff because it:

* matches realistic consumer behavior
* avoids asking users to run an offline calibration workflow
* still provides enough geometric structure for homography + scale recovery
* keeps the system fully classical-CV and on-device

Residual lens distortion on supported lenses is acceptable in V1 as long as resulting accuracy is good enough for accepted captures. Explicit per-device lens calibration can remain a future enhancement.

---

# 3. Reference Board Design

## 3.1 Chosen board type

The V1 printable reference should be a **ChArUco-style board**, not a plain checkerboard.

Definition:

* a checkerboard lattice
* with embedded ArUco markers inside selected squares
* producing uniquely indexable corners even when only part of the board is visible

Rationale:

* more robust than a plain checkerboard under partial occlusion
* easier to orient and index uniquely
* still classical CV
* better suited to “board next to object” than a full-background grid

## 3.2 Board geometry

V1 should ship with exactly one standard built-in board definition: `refboard_v1`.

Recommended geometry:

* board type: ChArUco
* square count: `11 x 8`
* square size: `15.0 mm`
* marker size: `11.0 mm`
* active board area: `165 mm x 120 mm`
* required quiet zone: `8 mm` white border on all sides
* total printable footprint with border and labels: sized to fit both US Letter and A4

This gives a board that is:

* small enough to print easily at home
* large enough to provide stable pose estimation
* wide enough in both axes to constrain homography better than a thin strip

## 3.3 Board asset requirements

The product should ship a printable board asset in at least:

* PDF
* SVG

The board asset must include:

* board ID text: `refboard_v1`
* a clear `TOP` label for humans
* a `print at 100%` instruction
* a printed verification bar, for example `50 mm`
* a small text note instructing the user not to scale-to-fit when printing

## 3.4 Physical usage guidance

The board is intended to be:

* printed once by the user
* cut out or left on the page
* placed beside the pattern piece in the same plane

The board should ideally be:

* printed on matte paper or cardstock
* kept flat
* optionally taped to a thin rigid backing if needed

The user workflow must remain simple. V1 should **not** require special calibration sessions, checkerboard datasets, or per-device setup.

## 3.5 Why not a plain checkerboard

A plain checkerboard is acceptable in principle, but it is not the preferred V1 board because:

* partial visibility makes corner indexing harder
* orientation is ambiguous
* robust board-to-template correspondence is weaker

If we are designing the printable ourselves, we should choose the more robust board.

# 4. Non-goals

The following are explicitly out of scope for V1:

* object segmentation
* SVG/path extraction
* per-user offline camera calibration
* support for arbitrary user-designed reference targets
* support for non-planar objects
* support for fisheye or ultra-wide lenses
* ML-based board detection
* multi-image self-calibration
* server-side processing

---

# 5. Product constraints

## 5.1 Capture assumptions

The input image is assumed to satisfy all of the following:

* shows the pattern piece and the printed reference board in the **same physical plane**
* the reference board is sufficiently visible to detect multiple markers/corners
* the image is reasonably sharp and not heavily motion-blurred
* the image is from a **main lens or telephoto-equivalent** only
* ultra-wide images are unsupported in V1
* the board is printed at the intended scale
* the board is approximately flat

## 5.2 Board placement assumptions

The reference board:

* may be beside the pattern piece, not necessarily underneath it
* does not need to fill the frame
* must not be heavily curled, folded, or warped
* should not be far away from the pattern piece in depth

The implementation should assume the board and pattern piece lie on the same plane. If they do not, the resulting rectification will be wrong.

## 5.3 Required user-provided inputs

V1 requires the following inputs at runtime:

* input image
* reference board spec ID or custom board spec path
* output target resolution, either:

  * pixels per millimeter, or
  * output width/height in pixels with scale derived automatically

Preferred V1 runtime contract:

* built-in default board: `refboard_v1`
* caller usually does **not** need to pass physical scale manually because it is encoded by the board definition

---

# 6. High-level workflow

The V1 pipeline is:

1. Load image
2. Validate the image for basic quality / capture suitability
3. Detect the printable reference board
4. Recover board corners / correspondences to canonical board coordinates
5. Estimate the plane homography from image space to board plane
6. Rectify the image to a fronto-parallel view
7. Recover metric scale from the known board geometry
8. Emit:

   * rectified image
   * transform metadata
   * debug overlays
   * quality metrics

Optional V1 enhancement:

* lightweight lens-distortion refinement from board residuals if it materially improves accuracy

However, this is not required for initial V1 acceptance.

---

# 7. Core architectural decision

## 7.1 Language

The core engine must be implemented in **Rust**.

Rationale:

* portable to CLI, mobile, and web
* no dependency on a server
* deterministic deployment
* strong performance on CPU
* easier eventual WASM/mobile reuse than Python

## 7.2 Compute model

V1 should be **CPU-first**.

Do not introduce GPU or WebGPU in V1.

## 7.3 Library boundaries

The codebase should be organized as:

* `rectify-core`
  Pure Rust library containing image geometry, board detection, homography, rectification, and metadata generation.

* `rectify-cli`
  Thin CLI wrapper around `rectify-core`.

Later frontends can wrap the same core:

* WASM frontend
* iOS binding
* Android binding

* `--fail-on-quality-warning`
* `--grayscale`

Exit codes:

* `0` success
* `1` fatal internal error
* `2` invalid input or arguments
* `3` reference board not found or invalid
* `4` capture quality failed
* `5` board detection failed
* `6` homography estimation failed

# 11. Internal pipeline specification

## 11.1 Stage A: Image ingestion

### Responsibilities

* load image
* apply EXIF orientation
* convert to internal image representation
* preserve original full resolution
* generate a working-resolution copy for analysis

### Suggested defaults

* analysis max dimension: `1600 px`
* final warp: full source resolution or bounded by memory limit

---

## 11.2 Stage B: Capture quality validation

This stage must reject obviously unsuitable inputs before attempting expensive geometry.

### Minimum checks required

#### Blur metric

Compute a blur/sharpness score, for example using variance of Laplacian or equivalent.

#### Exposure metric

Detect clipped highlights/shadows severe enough to impair board detection.

#### Board visibility estimate

Estimate whether enough board structure is visible.

Minimum requirement:

* enough candidate markers/corners to solve a stable homography
* detected points distributed across both board axes, not collapsed to a tiny region

#### Coverage metric

The engine should prefer captures where the board occupies enough image area to estimate stable geometry.

#### Lens mode validation

If EXIF or caller metadata indicates unsupported lens mode, reject or warn.

### Failure mode

If quality is insufficient, abort before rectification.

---

## 11.3 Stage C: Reference board detection

This is the most important estimation stage in V1.

### Objective

Detect the known printable board and recover image-to-board correspondences.

### Expected board properties

* known ChArUco layout
* uniquely identifiable markers
* subpixel-refinable checkerboard corners
* partial occlusion tolerated

### Required output of this stage

A fitted board observation sufficient to estimate the plane homography:

* detected marker IDs
* detected marker corners
* interpolated ChArUco corners or equivalent canonical correspondences
* confidence metrics

### Implementation guidance

V1 should use **classical CV**, not ML.

A practical approach is:

1. preprocess image for marker/corner visibility
2. detect candidate ArUco markers
3. decode marker IDs
4. reject inconsistent or low-confidence detections
5. interpolate ChArUco corners from marker detections
6. refine corner positions to subpixel accuracy

### Acceptable detectors

The exact detector is implementation-specific. Likely options include:

* OpenCV-style ArUco + ChArUco detection
* custom fiducial detector with equivalent outputs

The implementation should be robust to:

* different board/background colors around the board
* partial board visibility
* moderate perspective distortion

The implementation should not rely on the pattern piece itself having any regular grid.

### Board observation representation

Suggested internal representation:

```rust
struct BoardObservation {
    board_id: String,
    marker_ids: Vec<u32>,
    marker_corners_image: Vec<[Point2D; 4]>,
    charuco_corners_image: Vec<Point2D>,
    charuco_corners_board_mm: Vec<Point2D>,
    confidence: f64,
}
```

---

## 11.4 Stage D: Homography estimation

### Objective

Estimate a homography from the image plane to the canonical board plane.

### Required behavior

The estimated homography must satisfy:

* detected board corners map to their canonical metric board coordinates
* the board becomes fronto-parallel
* scale is directly tied to board millimeters

### Estimation strategy

Recommended approach:

1. solve an initial homography from image corners to canonical board corners
2. refine with robust nonlinear optimization minimizing board reprojection error
3. reject outliers and degenerate solutions

### Refinement objective

A robust refinement should minimize a weighted sum of:

* board corner reprojection error
* marker consistency error
* deviation from expected board geometry

### Failure condition

If no stable homography can be estimated, the pipeline must return failure code `6`.

---

## 11.5 Stage E: Metric scale recovery

### Objective

Use the known board square size to define rectified coordinates in metric units.

### Required input

* rectified board geometry
* `square_size_mm` from the board spec

### Output

Compute:

* `pixels_per_mm`
* `mm_per_pixel`

If caller specified `pixels_per_mm`, then the output image size is defined by that target scale.

### Canonical coordinate system

Define the rectified image coordinate system such that:

* board-space `x` and `y` align with the board axes
* units are millimeters in canonical board coordinates
* origin is tied to a documented board corner

---

## 11.6 Stage F: Output warp generation

### Objective

Generate the final rectified image.

### Requirements

* apply warp to the high-resolution image
* preserve as much valid image coverage as possible
* use interpolation appropriate for photographic input
* emit valid-region metadata

### Interpolation

Use a high-quality resampler, for example bilinear or bicubic.

### Output size

Output size may be determined by:

* transformed valid bounds at requested `pixels_per_mm`
* optional caller cap to prevent huge images

The output metadata must include:

* width
* height
* pixel scale
* valid bounds polygon if partial regions are undefined

# 14. Quality and rejection rules

V1 must be opinionated about refusing bad images.

## 14.1 Hard rejects

The engine should fail if any of the following are true:

* image unreadable
* board spec invalid or unsupported
* sharpness below threshold
* insufficient board evidence
* too few valid board correspondences
* homography fit unstable or degenerate
* output valid region too small to be useful

## 14.2 Soft warnings

The engine should warn but may continue if:

* board occupies a small area but detection is still stable
* board is partly occluded
* confidence is moderate, not high
* image is unusually oblique
* the board is near a frame edge where residual lens distortion may be larger

All warnings must be emitted in `quality.json`.

---

# 15. Acceptance criteria

The implementation is acceptable for V1 only if all of the following are true.

## 15.1 Functional acceptance

Given a valid image containing `refboard_v1`:

* the CLI completes successfully
* the rectified image is produced
* transform and quality metadata are emitted
* debug overlay is emitted

## 15.2 Geometric acceptance

Using a validation image set with known board geometry:

* rectified board lines should be visually straight
* rectified board corners should align to a consistent rectangular lattice
* recovered scale should be stable across the rectified image
* board reprojection error should remain below a predefined threshold

You should define this threshold during implementation testing, but V1 should target something like:

* median board-corner reprojection error under `1 px` at working resolution
* scale error under roughly `1-2%` on accepted images

## 15.3 Stability acceptance

For repeated images of the same plane under modest pose variation:

* scale should be stable
* output should not rotate/flip unpredictably
* quality scores should be reasonably consistent

## WASM Migration ✅ COMPLETE

Replaced Python/OpenCV detection backend with a pure-Rust implementation:

# 20. User-facing capture contract for V1

Even though this is an engine spec, the implementation assumes the following capture guidance exists for users:

* print `refboard_v1` once and reuse it
* keep the board in the same plane as the pattern piece
* keep the board visible in the photo
* avoid ultra-wide
* do not place the board or pattern piece too close to frame edge when possible
* capture in good diffuse light
* avoid motion blur
* keep the object and board flat
* avoid heavily reflective or glossy surfaces that obscure the board

The engine may enforce some of these through rejection logic.

# 22. Summary of required V1 behavior

V1 is a **single-photo, printable-reference-board rectification engine**.

It must:

* run locally
* require no server
* consume one image and a known printable board definition
* detect the board in-frame
* estimate the shared plane
* rectify the image into a fronto-parallel metric view
* emit images plus machine-readable metadata
* reject bad captures rather than silently producing misleading output

That is the complete scope.
