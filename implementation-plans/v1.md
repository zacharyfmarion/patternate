Below is a concrete **V1 technical spec** for the geometry-only phase of the product: **single input photo on a known square grid → undistorted + rectified image + scale metadata**.

This spec assumes someone else will implement it, so it is written as an engineering handoff, not a brainstorm.

---

# V1 Technical Spec: Single-Photo Grid Rectification Engine

## 1. Goal

Build an **on-device, single-photo rectification engine** that takes a handheld photo of a flat object placed on a **known square grid background** and returns a geometrically corrected top-down image suitable for later segmentation and vectorization.

V1 is **geometry only**. It does **not** need to detect or segment the fabric/object outline.

The engine must:

* remove lens distortion using a **precomputed camera calibration profile**
* estimate the plane of the grid in the image
* compute a fronto-parallel rectified view of that plane
* recover metric scale using the known grid pitch
* emit outputs and metadata for downstream processing

The implementation target is a **Rust core library** with a **CLI frontend**. The design must support later reuse in WebAssembly and native mobile apps.

---

# 2. Non-goals

The following are explicitly out of scope for V1:

* object segmentation
* SVG/path extraction
* support for uncalibrated cameras
* support for arbitrary non-square grids
* support for unknown grid pitch
* support for non-planar objects
* support for fisheye or ultra-wide lenses
* ML-based grid detection
* multi-image self-calibration
* server-side processing

---

# 3. Product constraints

## 3.1 Capture assumptions

The input image is assumed to satisfy all of the following:

* captured with a **camera/lens combination that has already been calibrated**
* shows a **planar square grid surface**
* the **physical grid pitch is known**
* the object lies approximately in the same plane as the grid
* there is enough visible grid around the object to estimate the plane
* the image is reasonably sharp and not heavily motion-blurred
* the image is from a **main lens or telephoto-equivalent** only
* ultra-wide images are unsupported in V1

## 3.2 Required user-provided inputs

V1 requires the following inputs at runtime:

* input image
* calibration profile for the camera/lens
* grid pitch in millimeters
* output target resolution, either:

  * pixels per millimeter, or
  * output width/height in pixels with scale derived automatically

---

# 4. High-level workflow

The V1 pipeline is:

1. Load image and calibration profile
2. Validate the image for basic quality / capture suitability
3. Undistort the image using the calibration profile
4. Detect the square grid structure in the undistorted image
5. Estimate the grid plane homography
6. Rectify the image to a fronto-parallel view
7. Compute scale using known grid pitch
8. Emit:

   * undistorted image
   * rectified image
   * calibration + transform metadata
   * debug overlays
   * quality metrics

---

# 5. Core architectural decision

## 5.1 Language

The core engine must be implemented in **Rust**.

Rationale:

* portable to CLI, mobile, and web
* no dependency on a server
* deterministic deployment
* strong performance on CPU
* easier eventual WASM/mobile reuse than Python

## 5.2 Compute model

V1 should be **CPU-first**.

Do not introduce GPU or WebGPU in V1. The performance bottleneck is not expected to justify the added complexity yet.

## 5.3 Library boundaries

The codebase should be organized as:

* `rectify-core`
  Pure Rust library containing image geometry, calibration, detection, rectification, and metadata generation.

* `rectify-cli`
  Thin CLI wrapper around `rectify-core`.

Later frontends can wrap the same core:

* WASM frontend
* iOS binding
* Android binding

# 6. Inputs and outputs

## 6.1 Primary inputs

### Input image

Supported formats:

* JPEG
* PNG

Requirements:

* RGB or grayscale
* EXIF orientation must be applied before processing

### Calibration profile

A structured file describing the calibrated camera intrinsics and distortion parameters for the exact device/lens mode.

Supported format:

* JSON

### Grid pitch

A required scalar:

* `grid_pitch_mm: f64`

Definition:

* physical distance in millimeters between adjacent grid intersections along one axis
* for a square grid, this equals the size of one square

Example:

* 1 inch grid → `25.4`
* 5 mm grid → `5.0`

### Output resolution request

One of:

* `pixels_per_mm`
* `pixels_per_grid_pitch`

Preferred V1 parameter:

* `pixels_per_mm`

---

## 6.2 Outputs

The engine must emit the following files:

### `undistorted.png`

The original image after lens distortion correction, with no planar rectification.

### `rectified.png`

The image warped so the grid plane is fronto-parallel and metric-consistent.

### `transform.json`

Machine-readable metadata describing:

* input dimensions
* undistortion parameters used
* homography
* output scale
* output pixel-to-mm mapping
* quality/confidence metrics

### `debug_overlay.png`

Debug image showing detected grid evidence and fit diagnostics.

### `quality.json`

Detailed quality metrics and warnings.

# 7. Calibration profile specification

## 7.1 Distortion model

Use a standard Brown–Conrady camera model for V1.

Parameters:

* focal lengths: `fx`, `fy`
* principal point: `cx`, `cy`
* radial distortion: `k1`, `k2`, `k3`
* tangential distortion: `p1`, `p2`

Optional but not required in V1:

* `k4+`
* thin prism terms
* rolling shutter correction

## 7.2 Calibration profile JSON schema

Example:

```json
{
  "schema_version": 1,
  "camera_id": "iphone15pro_main_1x",
  "image_size_px": {
    "width": 4032,
    "height": 3024
  },
  "intrinsics": {
    "fx": 2850.12,
    "fy": 2847.91,
    "cx": 2012.43,
    "cy": 1510.88
  },
  "distortion": {
    "model": "brown_conrady",
    "k1": -0.1123,
    "k2": 0.0412,
    "k3": -0.0087,
    "p1": 0.0009,
    "p2": -0.0006
  },
  "supported_zoom_mode": "1x",
  "notes": "Main lens only"
}
```

## 7.3 Calibration profile constraints

The engine must reject a profile if:

* schema version is unsupported
* image size is incompatible with the input without a defined scaling rule
* required parameters are missing
* distortion model is unsupported

## 7.4 Resolution scaling rule

If the input image resolution differs from the calibration profile’s nominal image size but has the same aspect ratio, the intrinsics must be scaled proportionally.

If aspect ratio differs, the engine must reject the input unless explicit crop/resize metadata exists.

## 8.1 Command

Proposed CLI:

```bash
rectify-cli rectify \
  --input input.jpg \
  --calibration iphone15pro_main_1x.json \
  --grid-pitch-mm 25.4 \
  --pixels-per-mm 10 \

```

## 8.2 Required flags

* `--input <path>`
* `--calibration <path>`
* `--grid-pitch-mm <float>`
* `--output-dir <path>`

## 8.3 Optional flags

* `--pixels-per-mm <float>`
* `--max-processing-dim <int>`
* `--debug`
* `--fail-on-quality-warning`
* `--grayscale`
* `--json-stdout`

## 8.4 Exit behavior

Exit codes:

* `0` success
* `1` fatal internal error
* `2` invalid input or arguments
* `3` calibration/profile mismatch
* `4` capture quality failed
* `5` grid detection failed
* `6` homography estimation failed

# 9. Internal pipeline specification

## 9.1 Stage A: Image ingestion

### Responsibilities

* load image
* apply EXIF orientation
* convert to internal image representation
* preserve original full resolution
* generate a working-resolution copy for analysis

### Requirements

* processing should happen on a downscaled copy for detection
* final warp should happen on the highest useful resolution available

### Suggested defaults

* analysis max dimension: `1600 px`
* final warp: full source resolution or bounded by memory limit

---

## 9.2 Stage B: Capture quality validation

This stage must reject obviously unsuitable inputs before attempting expensive geometry.

### Minimum checks required

#### Blur metric

Compute a blur/sharpness score, for example using variance of Laplacian or equivalent.

The exact metric may vary, but must produce a normalized score in `quality.json`.

#### Exposure metric

Detect clipped highlights/shadows severe enough to impair line detection.

#### Grid visibility estimate

Estimate whether enough grid structure is visible in the image.

Minimum requirement:

* both principal grid directions must be observable
* enough candidate grid evidence must exist away from the central object occlusion

#### Border coverage

The engine should prefer images where grid evidence exists near multiple image sides. Low border coverage should downgrade confidence or fail.

#### Lens mode validation

If EXIF or caller metadata indicates unsupported lens mode, reject or warn.

### Output

This stage writes structured metrics to `quality.json`.

### Failure mode

If quality is insufficient, abort before rectification.

---

## 9.3 Stage C: Undistortion

### Input

* original image
* calibration profile

### Operation

Apply Brown–Conrady undistortion to the image.

The undistortion operation must support:

* point mapping
* full-image remap
* reuse of precomputed maps if resolution repeats

### Deliverables

* undistorted working image
* undistorted full-resolution image or remap instructions for later application

### Notes

Do not crop aggressively in V1. Preserve the maximal valid image region and record valid bounds in metadata.

---

## 9.4 Stage D: Grid detection

This is the most important estimation stage in V1.

### Objective

Recover the 2D lattice structure of the square grid in the undistorted image.

### Expected grid properties

* two dominant families of approximately straight, approximately parallel lines
* the two families are approximately orthogonal in the physical scene
* line spacing is uniform in the physical plane
* object may occlude central region

### Required output of this stage

A fitted grid model sufficient to estimate the plane homography:

* dominant line family A
* dominant line family B
* candidate intersections or equivalent lattice constraints
* confidence metrics

### Implementation guidance

V1 should use **classical CV**, not ML.

A practical approach is:

1. preprocess image for line visibility
2. detect candidate line segments
3. cluster segments into two dominant directional families
4. fit long-support lines for each family
5. infer lattice spacing and intersection structure
6. reject outliers using RANSAC or robust fitting

### Acceptable detectors

The exact detector is implementation-specific. Likely options include:

* line segment detector
* edge + Hough-based line extraction
* directional filtering + peak analysis

The implementation should be robust to varying grid and background colors and must not rely on fixed RGB or grayscale thresholds that assume a specific checkerboard color scheme. Detection should be driven by edge/lattice structure, ideally using color-aware edge evidence when luminance contrast alone is weak.

The implementation should not assume the grid is fully visible. It must tolerate partial occlusion by the object.

### Grid model representation

Suggested internal representation:

```rust
struct GridModel {
    family_a: Vec<Line2D>,
    family_b: Vec<Line2D>,
    intersections: Vec<Point2D>,
    spacing_a_px_projective: Vec<f64>,
    spacing_b_px_projective: Vec<f64>,
    confidence: f64,
}
```

This does not need to expose raw spacing as final metric scale yet. It is just the intermediate fitted structure.

---

## 9.5 Stage E: Homography estimation

### Objective

Estimate a homography from the undistorted image plane to a canonical fronto-parallel grid plane.

### Required behavior

The estimated homography must satisfy:

* mapped grid line families become orthogonal
* lattice spacing becomes uniform
* the plane is rectified up to known scale

### Mathematical target

We want a projective transform `H` such that points on the observed grid plane map to a rectified plane coordinate system where:

* x and y axes align with the grid
* distances correspond to real-world grid pitch

### Estimation strategy

Recommended approach:

1. fit vanishing directions from the two line families
2. derive an initial affine/metric rectification
3. refine with nonlinear optimization using lattice consistency loss

### Refinement objective

A robust refinement should minimize a weighted sum of:

* line reprojection error
* intersection reprojection error
* deviation from orthogonality
* nonuniform spacing in rectified coordinates

### Failure condition

If no stable homography can be estimated, the pipeline must return failure code `6`.

---

## 9.6 Stage F: Metric scale recovery

### Objective

Use the known grid pitch to define rectified coordinates in metric units.

### Required input

* estimated rectified grid spacing in rectified pixel coordinates
* `grid_pitch_mm`

### Output

Compute:

* `pixels_per_mm`
* `mm_per_pixel`

If caller specified `pixels_per_mm`, then the output image size is defined by that target scale.

### Canonical coordinate system

Define the rectified image coordinate system such that:

* +x is along one grid axis
* +y is along the other grid axis
* origin is arbitrary but stable within the output metadata

The exact origin placement does not matter for V1 as long as it is consistent and documented.

---

## 9.7 Stage G: Output warp generation

### Objective

Generate the final rectified image.

### Requirements

* apply warp to high-resolution image
* preserve as much valid image coverage as possible
* use interpolation appropriate for photographic input
* emit valid region metadata

### Interpolation

Use a high-quality resampler, for example bilinear or bicubic. Nearest-neighbor is not acceptable for the photographic rectified image.

### Output size

Output size may be determined by:

* transformed valid bounds at requested `pixels_per_mm`
* optional caller cap to prevent huge images

The output metadata must include:

* width, height
* pixel scale
* valid bounds polygon if partial regions are undefined

# 12. Quality and rejection rules

V1 must be opinionated about refusing bad images. This is necessary for a production workflow.

## 12.1 Hard rejects

The engine should fail if any of the following are true:

* image unreadable
* calibration profile invalid or incompatible
* sharpness below threshold
* insufficient grid evidence
* only one dominant grid direction detectable
* homography fit unstable or degenerate
* output valid region too small to be useful

## 12.2 Soft warnings

The engine should warn but may continue if:

* grid visibility near border is weak
* object occludes a large central area but enough border grid remains
* confidence is moderate, not high
* image is unusually oblique

All warnings must be emitted in `quality.json`.

---

# 13. Acceptance criteria

The implementation is acceptable for V1 only if all of the following are true.

## 13.1 Functional acceptance

Given a valid calibrated image of a square grid plane:

* the CLI completes successfully
* the undistorted image is produced
* the rectified image is produced
* transform and quality metadata are emitted
* debug overlay is emitted

## 13.2 Geometric acceptance

Using a validation image set with known grid pitch:

* rectified grid lines should be visually straight
* rectified grid intersections should form a consistent square lattice
* recovered scale should be stable across the rectified image
* local spacing error should remain below a predefined threshold

You should define this threshold during implementation testing, but V1 should target something like:

* median grid-spacing error under `1%`
* no gross local warping in accepted images

## 13.3 Stability acceptance

For repeated images of the same plane under modest pose variation:

* scale should be stable
* output should not rotate/flip unpredictably
* quality scores should be reasonably consistent

# 17. Calibration workflow specification

Because V1 requires pre-calibrated cameras, calibration must be treated as part of the system.

## 17.1 V1 calibration scope

Calibration is out of runtime scope, but the implementation assumes it exists.

Each calibration profile corresponds to:

* one device model
* one specific lens mode
* one nominal image resolution/aspect ratio

## 17.2 Supported calibration concept

A separate calibration utility or offline process produces the calibration JSON.

That tool is not part of this V1 spec, but the engine must be designed to consume its outputs cleanly.

## 17.3 Practical rule for V1

Do not try to generalize one profile across:

* different lenses
* ultra-wide vs main
* materially different crop modes

Treat those as separate calibrations.

# 18. User-facing capture contract for V1

Even though this is an engine spec, the implementation assumes the following capture guidance exists for users:

* use supported calibrated lens only
* avoid ultra-wide
* include visible grid around all sides of the object
* do not place object too close to frame edge
* capture in good diffuse light
* avoid motion blur
* keep the object flat on the grid
* do not use heavily reflective or glossy surfaces that obscure grid lines

The engine may enforce some of these through rejection logic.

V1 is a **single-photo, pre-calibrated, square-grid rectification engine**.

It must:

* run locally
* require no server
* consume one image, one calibration profile, and known grid pitch
* undistort the image
* estimate the grid plane
* rectify the image into a fronto-parallel metric view
* emit images plus machine-readable metadata
* reject bad captures rather than silently producing misleading output
